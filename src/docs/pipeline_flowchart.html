Phase 4 归一化注册表详情 (NormalizationRegistry)

📘 静态注册表 (YAML 配置)

  • ModalityRegistry — RFC 义务词归一化
    MUST → must, SHOULD → should, MAY → may...
  • RoleRegistry — LLM 角色名 → Maude ActorType
    "DNS server" → Server, "resolver" → Resolver
  • EventPatternRegistry — LLM 事件 → Maude event pattern
    "receives query" → recv_query
  • ActionOpsRegistry — LLM 动作 → Maude action type
    "sends response" → send_message

📗 动态注册表 (maude_contract.json)

  • PredicateRegistry — Guard predicates
    从 rules[].guard_slots 动态加载模板
  • AttributeRegistry — Actor 状态属性
    从 actors[].state_interface 加载 sort/mode
  • contract 由 maude_parser/ 从已有 Maude 源码提取生成

🔄 三层归一化流程

  • L1 RuleBasedNormalizer — 确定性别名匹配 (子串匹配, 长模式优先)
  • → unresolved items — 无法匹配的项收集到列表
  • L2 LLMAssistedNormalizer — 对 unresolved 项调用 LLM
    产出 MaudeProposal (strategy: reuse_existing / new_operator / new_sort / skip)
  • L3 人工审核 — proposals 保存为 JSON 文件供审核

📐 Semantic IR 格式 (ECA)

  • id — 规则唯一标识
  • modality — MUST / SHOULD / MAY / MUST_NOT
  • actor — { name, type }
  • event — { kind: recv|send|timeout|..., expr }
  • conditions[] — { kind: guard|state_check|..., expr }
  • actions[] — { kind: send|state_update|..., expr }
  • provenance — { document, section, anchor, source_text }
